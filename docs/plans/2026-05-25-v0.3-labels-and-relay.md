# v0.3 Implementation Plan — Audit Labels + Webhook Relay

**Date:** 2026-05-25
**Status:** Plan, pending execution
**Target version:** purple-wolf v0.3.0 (workspace) + purple-wolf-relay 0.1.0
**Predecessor:** v0.2.0 (header inspection, bounded rate-limiter, XFF trust model, honest CRS measurements)

---

## 1. Summary

v0.3 ships two coordinated additions:

1. **Audit labels** in `purple-wolf-core` — a free-form `labels: map<string,string>` field on the Middleware plugin config, echoed into every audit-log entry. Tiny change in the WAF; gives operators a stable way to attach service/tenant/owner/environment metadata to detection events without forking the audit schema.
2. **`purple-wolf-relay`** — a new standalone, vendor-neutral service that tails Traefik's log stream, parses purple-wolf audit lines, applies optional enrichment, and fans out HMAC-signed HTTP webhooks to operator-configured subscribers with at-least-once delivery, retries, and a dead-letter queue.

Both are designed to compose: the WAF stays focused on detection, the relay stays focused on delivery. Neither knows about Guara Cloud, ACME, or any other tenant taxonomy — operators describe their world through opaque labels.

**Why now:**
- Adopters consistently want push-based event consumption (Slack alerts, SIEM ingest, per-tenant webhook subscribers) on top of the pull-based log-pipeline model that v0.2 supports.
- A separate relay is the right shape: the http-wasm sandbox has no outbound-HTTP capability, and putting webhook delivery in the WAF would couple a security-critical hot path to a notoriously failure-prone subsystem (HTTP delivery to third parties).
- Labels are a v0.3 prerequisite because without per-event routing metadata, subscribers have to parse Traefik's middleware name to figure out which tenant an event belongs to.

**Why v0.3 (semver):**
- `core::Config` gains a new optional field — additive, default-empty, backward-compatible with v0.2 configs. Technically a minor bump.
- New crate `purple-wolf-relay` 0.1.0 starts at pre-1.0 explicitly.
- Middleware schema bumped from `purpleWolf:1` to `purpleWolf:1` (no major change) — `labels` is `#[serde(default)]` so old configs continue to parse.

### Goals

- Operators can attach arbitrary `key=value` labels per Middleware; labels appear verbatim in every audit-log line for that Middleware.
- A standalone `purple-wolf-relay` binary reads Traefik logs, parses purple-wolf audit JSON, and delivers HMAC-signed webhooks to filter-matched subscribers.
- A documented, stable webhook protocol (`purple-wolf.audit/v1` envelope) that any subscriber in any language can implement against.
- At-least-once delivery with per-event idempotency keys, exponential backoff, and dead-letter queue.
- Prometheus metrics and structured logs for the relay.
- Zero coupling to any tenant model: labels are opaque, subscribers' filters work on label-key equality and rule-pattern globs.

### Non-goals (v0.3)

- **Long-term event storage.** The relay is a delivery service, not a SIEM. DLQ is bounded.
- **Subscriber UI / management console.** Config is YAML/JSON files; building a UI on top is a separate product.
- **Per-tenant rate-limiting *at the relay*.** Subscriber-side bottlenecks are the subscriber's problem; the relay enforces per-subscriber concurrency only.
- **Transformation pipelines.** Labels go through verbatim; if you need to reshape, do it at the subscriber or upstream of the relay with Vector/Fluent Bit.
- **Multi-tenant SaaS isolation.** Each deployment runs its own relay instance with its own config; tenants are label values, not first-class isolation boundaries.
- **Streaming protocols** (WebSocket, gRPC, SSE). Pure HTTP POST webhooks for v0.3.
- **Source plugins beyond log-tail and stdin.** Kafka / Loki / NATS sources are v0.4+.

### 2.3 Label schema (operator contract)

- **Key constraint:** matches `^[a-z][a-z0-9_.-]{0,62}$` (lowercase ASCII, dot-separated, max 63 chars per key — OpenTelemetry resource-attribute style).
- **Value constraint:** any UTF-8 string up to 1024 bytes; control characters (`\x00..=\x1F`, `\x7f`) stripped by the WAF before serialization (same scrub as `blocked_detail`, see NEW-I1).
- **Quantity cap:** ≤ 32 keys, ≤ 4096 total bytes across all keys+values. Exceeding either is a parse error.
- **Reserved namespace:** keys starting with `purple_wolf.` are reserved for fields the WAF or relay sets (`purple_wolf.middleware`, `purple_wolf.router`); user labels in user-defined namespace. Operator-supplied `purple_wolf.*` keys are silently dropped with a one-time warn log per Middleware instance.

### 2.4 Webhook envelope (subscriber contract)

```json
{
  "schema": "purple-wolf.audit/v1",
  "event_id": "01HXYZ...",            // ULID; stable across retries
  "delivery_id": "01HXYZ...",          // new per attempt
  "delivered_at": "2026-05-25T17:30:02.123Z",
  "attempt": 1,                        // 1-based; matches X-PurpleWolf-Attempt
  "labels": { "<key>": "<value>", … },
  "source": {
    "middleware": "strict-waf",        // from Traefik's log enrichment
    "router": "checkout",
    "entry_point": "web",
    "relay_instance": "relay-abc-1"    // for cluster-wide dedup
  },
  "event": {                            // verbatim audit fields
    "host": "checkout.acme.example",
    "path": "/api/v1/cart",
    "query": "id=1%27+OR+%271%27%3D%271",
    "method": "POST",
    "source_ip": "203.0.113.7",
    "action": "block",
    "blocked_rule": "injection/sqli",
    "blocked_severity": "critical",
    "blocked_detail": "SQLi in field: ...",
    "would_block_rules": ["reputation/rate_limited"]
  }
}
```

HTTP headers on every POST:

```
Content-Type: application/json
User-Agent: purple-wolf-relay/<version>
X-PurpleWolf-Event-Id: 01HXYZ...
X-PurpleWolf-Delivery-Id: 01HXYZ...
X-PurpleWolf-Schema: purple-wolf.audit/v1
X-PurpleWolf-Signature: sha256=<hex HMAC of body>
X-PurpleWolf-Attempt: 1
X-PurpleWolf-Timestamp: 1748194202   (unix seconds; signed alongside body to prevent replay)
```

HMAC computation: `HMAC-SHA256(secret, timestamp + "." + body_bytes)` → hex-encoded. Subscribers MUST reject if timestamp skew > 5 minutes (configurable hint in docs).

## 3. Risk register

| Risk | Impact | Likelihood | Mitigation |
|---|---|---|---|
| Audit-log line size grows past Loki/Promtail limits (~64 KiB) when labels are large | Logs silently truncated; webhooks malformed | Low | Cap labels at 4 KiB total (validated at parse time); document the budget in `configuration.md`. |
| Webhook delivery storm (subscriber endpoint returns 500 to every event) saturates relay memory | Relay OOM | Medium | Per-subscriber bounded mpsc queue (default 10k); reject new events for that subscriber when full, emit a `dropped` metric, continue serving other subscribers. |
| Subscriber secret leaked in config | Compromise of HMAC | Medium | Secrets are env-var-references or file-references only; never inline. Document with a "don't commit your config" note. |
| Log-tail source loses events on rotation | At-least-once becomes "at-most-once" | Low | Use `notify` crate + position bookmark file written every N events; reopen file on rotation, resume from bookmark. |
| Relay version mismatch with WAF audit schema | Envelope wraps unknown fields, subscribers confused | Low | Envelope is forward-compatible: unknown fields under `event.*` pass through; `schema` field changes only on breaking change. Versioning policy documented. |
| HMAC timestamp skew rejects valid deliveries | Subscriber drops legitimate events | Medium | 5-minute default skew window; use NTP-synced clocks; document in `webhook-protocol.md`. |
| A misbehaving enricher (HTTP lookup, slow upstream) blocks the pipeline | Backpressure stalls the whole relay | Medium | Enrichers run with a per-call timeout (default 500ms); on timeout the event proceeds without that enrichment, emit `enricher_timeout` metric. |
| The plugin's labels field gets misused for high-cardinality data (e.g. user_id) | Audit log + relay metrics cardinality explodes; Prometheus storage cost grows | High | Documentation explicitly warns against high-cardinality labels (no `user_id`, no per-request UUIDs); per-label key budget enforced at WAF; relay metrics use a label-cardinality cap. |

### Task 1: Add `labels` to `core::Config` with validation

**Files:**
- Modify: `crates/purple-wolf-core/src/config.rs`

**Steps:**

1. Import `std::collections::BTreeMap` (not `HashMap` — BTreeMap iterates in key order, which gives deterministic audit-log JSON).
2. Add the field with serde defaults and explicit cap constants:

   ```rust
   /// Maximum number of label keys per Middleware. See v0.3 plan §2.3.
   pub const MAX_LABEL_KEYS: usize = 32;
   /// Maximum total bytes summed across all key+value pairs. See plan §2.3.
   pub const MAX_LABEL_BYTES: usize = 4096;
   /// Maximum byte length of a single label value.
   pub const MAX_LABEL_VALUE_BYTES: usize = 1024;
   ```

3. On `Config`, append:

   ```rust
   /// Operator-supplied labels echoed into every audit log line for
   /// this Middleware. Opaque to the WAF; consumed by downstream log
   /// pipelines or `purple-wolf-relay` (v0.3) for routing/filtering.
   ///
   /// Keys must match `^[a-z][a-z0-9_.-]{0,62}$`. Keys starting with
   /// `purple_wolf.` are reserved for fields the WAF/relay set.
   /// At most `MAX_LABEL_KEYS` keys and `MAX_LABEL_BYTES` total bytes;
   /// per-value bytes capped at `MAX_LABEL_VALUE_BYTES`.
   #[serde(default)]
   pub labels: std::collections::BTreeMap<String, String>,
   ```

4. Add a public validator:

   ```rust
   /// Validate the label map against the documented caps. Returns
   /// `Err(message)` describing the first violation; messages name the
   /// offending key so a tenant can fix their YAML immediately.
   pub fn validate_labels(labels: &BTreeMap<String, String>) -> Result<(), String> {
       if labels.len() > MAX_LABEL_KEYS {
           return Err(format!(
               "labels: {} keys exceeds maximum {MAX_LABEL_KEYS}", labels.len()
           ));
       }
       let mut total = 0usize;
       for (k, v) in labels {
           if !is_valid_label_key(k) {
               return Err(format!(
                   "labels: key {k:?} does not match ^[a-z][a-z0-9_.-]{{0,62}}$"
               ));
           }
           if v.len() > MAX_LABEL_VALUE_BYTES {
               return Err(format!(
                   "labels: value for key {k:?} is {} bytes; max {MAX_LABEL_VALUE_BYTES}",
                   v.len()
               ));
           }
           total = total.saturating_add(k.len() + v.len());
       }
       if total > MAX_LABEL_BYTES {
           return Err(format!("labels: total size {total} bytes exceeds {MAX_LABEL_BYTES}"));
       }
       Ok(())
   }

   fn is_valid_label_key(k: &str) -> bool {
       let bytes = k.as_bytes();
       if bytes.is_empty() || bytes.len() > 63 { return false; }
       if !bytes[0].is_ascii_lowercase() { return false; }
       bytes[1..].iter().all(|b|
           b.is_ascii_lowercase() || b.is_ascii_digit() || matches!(b, b'_'|b'.'|b'-'))
   }
   ```

5. Scrub user-supplied values for ASCII control characters (mirror `truncate_bytes` from `injection.rs`); reuse the same logic factored into a helper, or inline. Keep this in `Config::validate_labels` OR apply at audit-line build time — pick the latter so validation stays cheap and pure.

### Task 3: Audit log includes labels

**Files:**
- Modify: `crates/purple-wolf-core/src/audit.rs`
- Modify: `crates/purple-wolf-core/src/detectors/`... (no, only audit; detectors don't see labels)

**Steps:**

1. Extend `AuditEntry` with the field:

   ```rust
   /// Operator-supplied labels from Config.labels. Emitted as-is so
   /// downstream relays/log pipelines can route on them.
   #[serde(skip_serializing_if = "BTreeMap::is_empty")]
   pub labels: std::collections::BTreeMap<String, String>,
   ```

   `skip_serializing_if` keeps v0.2-shape logs (no labels field) when nothing is configured — minimum churn for existing log queries.

2. Add a constructor variant or extend the existing one:

   ```rust
   impl AuditEntry {
       pub fn from_with_labels(
           req: &Request,
           decision: &Decision,
           labels: &BTreeMap<String, String>,
       ) -> AuditEntry {
           let mut entry = AuditEntry::from(req, decision);
           // Strip ASCII control chars from values before emit, same scrub as
           // detectors/injection.rs::truncate_bytes — labels are operator-set
           // but treating them as opaque means we still apply the safety pass.
           entry.labels = labels.iter()
               .map(|(k, v)| (k.clone(), scrub_value(v)))
               .collect();
           entry
       }
   }

   fn scrub_value(v: &str) -> String {
       v.chars()
           .map(|c| if (c as u32) < 0x20 || c == '\x7f' { '.' } else { c })
           .collect()
   }
   ```

3. Keep the existing `AuditEntry::from(req, decision)` for callers that don't have labels (preserves v0.2 API).

4. Update `purple-wolf-traefik/src/entry.rs`: replace the `AuditEntry::from(&req, &decision)` call with `AuditEntry::from_with_labels(&req, &decision, &cfg.labels)`.

### Schema

| Constraint | Value |
|---|---|
| Key regex | `^[a-z][a-z0-9_.-]{0,62}$` (lowercase ASCII, OpenTelemetry-style) |
| Value | any UTF-8 up to 1024 bytes; control chars stripped at serialize time |

### Task 6: Write `docs/webhook-protocol.md`

**Files:**
- Create: `docs/webhook-protocol.md`

**Steps:**

The document MUST cover, in this order:

1. **Audience & scope** — "this is for subscriber implementers; the relay sends, the subscriber receives".
2. **Envelope schema v1** — full JSON shape, every field described (name, type, required/optional, semantics, example).
3. **HTTP semantics:**
   - Method: `POST`
   - `Content-Type: application/json`
   - Required request headers (table)
   - Subscriber response: 2xx = delivered (relay marks done); 3xx = relay does NOT follow redirects (treat as failure); 408/429/5xx = retryable; 4xx (except 408/429) = permanent failure, send to DLQ.
4. **HMAC verification:**
   - Algorithm: HMAC-SHA256
   - Signed payload: `<X-PurpleWolf-Timestamp>.<request body bytes>` — timestamp prevents replay
   - Header format: `X-PurpleWolf-Signature: sha256=<lowercase hex>`
   - Constant-time comparison required on the subscriber side
   - Recommended skew tolerance: 5 minutes
5. **Idempotency:**
   - `event_id` is stable across retries; subscribers MUST dedupe on it
   - `delivery_id` changes per attempt (useful for subscriber-side logging only)
   - Recommended dedup window: 24 hours
6. **Retry semantics from the relay's side** (informational for subscribers, NOT contractual):
   - Exponential backoff with jitter, base 500ms, max 10m, default 8 attempts
   - Subscribers should not assume any specific retry schedule
7. **Versioning policy:**
   - `schema` field in the envelope is `purple-wolf.audit/<MAJOR>`
   - New optional fields can appear at any minor version without bumping the schema number
   - Removed fields or semantic changes bump the schema number; relay can be configured to deliver both versions during a migration window
8. **Reference subscriber implementations:**
   - Python (Flask): ~30 lines
   - Go (net/http): ~50 lines
   - TypeScript (Hono / Express): ~30 lines
   - Each MUST demonstrate: signature verification, timestamp skew check, event_id dedup (in-memory LRU is fine for the example), idempotent processing, 200 on success, 5xx on transient error.
9. **Error handling table** — how the relay interprets each status class
10. **Security checklist for subscribers:**
    - Always verify signature (don't trust source IP)
    - Always check timestamp skew (replay protection)
    - Always dedupe on event_id (at-least-once → exactly-once at subscriber)
    - Subscriber endpoint should be idempotent — even with dedup, replays can happen
    - Subscriber endpoint should respond within 30s (relay timeout default)
    - Secret rotation: subscribers must support overlap window (both old and new HMAC accepted) during rotation

**Sample subscriber (Python — to be included verbatim in the doc):**

```python
import hashlib, hmac, json, os, time
from collections import OrderedDict
from flask import Flask, request, abort

SECRET = os.environ["PURPLEWOLF_SECRET"].encode()
SKEW_S = 300         # 5 min
SEEN = OrderedDict() # event_id → ts; LRU; replace with Redis in prod
SEEN_CAP = 10_000

app = Flask(__name__)

@app.post("/webhook")
def receive():
    ts = request.headers.get("X-PurpleWolf-Timestamp", "")
    sig = request.headers.get("X-PurpleWolf-Signature", "")
    eid = request.headers.get("X-PurpleWolf-Event-Id", "")
    if not (ts.isdigit() and sig.startswith("sha256=") and eid):
        abort(400)
    if abs(time.time() - int(ts)) > SKEW_S:
        abort(401)  # replay protection
    body = request.get_data()
    expected = "sha256=" + hmac.new(
        SECRET, f"{ts}.".encode() + body, hashlib.sha256
    ).hexdigest()
    if not hmac.compare_digest(expected, sig):
        abort(401)
    if eid in SEEN:
        return ("", 200)  # dedup; idempotent ack
    SEEN[eid] = time.time()
    if len(SEEN) > SEEN_CAP:
        SEEN.popitem(last=False)
    event = json.loads(body)
    # ... your handler ...
    return ("", 200)
```

```rust
use serde::{Deserialize, Serialize};
use std::collections::BTreeMap;

pub const SCHEMA_V1: &str = "purple-wolf.audit/v1";

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct Envelope {
    pub schema: String,
    pub event_id: String,      // ULID; stable across retries
    pub delivery_id: String,   // ULID; new per attempt
    pub delivered_at: chrono::DateTime<chrono::Utc>,
    pub attempt: u32,
    #[serde(default, skip_serializing_if = "BTreeMap::is_empty")]
    pub labels: BTreeMap<String, String>,
    pub source: EnvelopeSource,
    pub event: serde_json::Value,  // pass through audit JSON verbatim
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct EnvelopeSource {
    pub middleware: Option<String>,
    pub router: Option<String>,
    pub entry_point: Option<String>,
    pub relay_instance: String,
}

impl Envelope {
    /// Build a fresh envelope for a parsed audit event. event_id is stable;
    /// delivery_id rolls per attempt via `Envelope::with_attempt(n)`.
    pub fn new(
        event: serde_json::Value,
        source: EnvelopeSource,
        labels: BTreeMap<String, String>,
    ) -> Self {
        let id = ulid::Ulid::new().to_string();
        Self {
            schema: SCHEMA_V1.into(),
            event_id: id.clone(),
            delivery_id: id,
            delivered_at: chrono::Utc::now(),
            attempt: 1,
            labels,
            source,
            event,
        }
    }
    pub fn with_attempt(mut self, n: u32) -> Self {
        self.delivery_id = ulid::Ulid::new().to_string();
        self.delivered_at = chrono::Utc::now();
        self.attempt = n;
        self
    }
}
```

# Phase F — Subscriber delivery

**Goal:** events flow from the parser through each subscriber's filter, are HMAC-signed, and POSTed with retries + DLQ.

use hmac::{Hmac, Mac};
use sha2::Sha256;

pub struct Signer { secret: zeroize::Zeroizing<Vec<u8>> }

impl Signer {
    pub fn new(secret: impl Into<Vec<u8>>) -> Self {
        Self { secret: zeroize::Zeroizing::new(secret.into()) }
    }
    /// Sign `timestamp_secs + "." + body` per docs/webhook-protocol.md.
    /// Returns the value of the `X-PurpleWolf-Signature` header

        mac.update(body);

    }
}

### Task 20: Pipeline integration — wire sources → parser → enrichers → subscribers

**Files:**
- Modify: `crates/purple-wolf-relay/src/pipeline.rs`

**Steps:**

1. For each source: spawn task; sources push `RawEvent` to a single `tokio::sync::mpsc` (capacity 1024).

- Drain the parser → fan-out pipeline.

## 6. Open questions

These are decisions deliberately deferred to implementation time. None block starting Phase A.

1. **Persistence layer for the DLQ.** In-memory is fine for v0.3. Add SQLite in v0.4 OR adopt a small embedded option like `sled`? Decision before v0.4 planning.
2. **Replay UX.** v0.3 admin endpoint `POST /dlq/<id>/replay` replays all DLQ entries. Should it support time-window replay (`?since=T1&until=T2`)? Add when first operator asks.
3. **Subscriber-supplied filter expressions vs. structured filters.** v0.3 ships structured (labels + severity_min + glob). v0.4 might add a tiny expression language (`labels.tenant == 'acme' && event.action == 'block' && severity >= high`). Defer; revisit when filter complexity outgrows structured.
4. **Relay clustering / HA.** v0.3 is single-instance per Traefik deployment. v0.4+ could shard by subscriber_id or by event_id hash. Today, run two relays consuming the same log stream + per-instance event_id dedup at the subscriber side — operationally workable.
5. **Outbound queues other than HTTP.** Should we support Kafka / NATS sink natively, or punt to "run Vector after the relay"? Punt for v0.3; revisit only with concrete demand.
6. **Auth on the admin endpoints** (`/metrics`, `/readyz`, `/dlq/*`). v0.3: bind to internal network only; v0.4: optional bearer-token via config.
